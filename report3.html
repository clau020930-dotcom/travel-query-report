去哪玩 — AIM Query & Session 分析报告

数据范围：2026/6/23 - 6/29（7天） 底表：dm_dots_query_distribution_detail_1d_di + query 抽样标注 生成日期：2026/7/3

结论先行

1. 旅游是 AIM 天然的高价值场景，但用户心智尚未建立
旅游 query 占 AIM 整体仅 2.06%（已分类 3.68%），远低于个人成长(20.3%)和生活方式大盘(16.7%)。但旅游用户的意图极其明确 —— 77.7% 是 Do+Decide（做计划+要推荐），是所有场景中"目的性最强"的用户群。问题不在能力，在于用户还没想到来点点做旅游规划。
2. 旅游是 AIM 多轮对话的标杆场景，远超购物
旅游 session 多轮率 77.4%（≥2轮），10+轮深度对话占 15.5%（段内均值 18.4 轮）。对比购物场景 80.5% 只有 2 轮 —— 旅游用户会持续追问细节、修改行程、比较方案，这是 AI Chat 相比搜索引擎最大的差异化价值。
3. 行程规划是绝对核心需求，占 41.2%
对 3,396 个旅游 session 的 LLM 标注分析显示，行程规划大类以 1,400 条（41.2%）排名第一。其中 L2-1 综合行程规划 846 条（24.9%）远超第二名景点推荐 601 条。用户 query 极长极具体（平均 107 字），包含日期、人数、预算、约束条件 —— 这是搜索引擎做不到的事情。
4. 用户已经在把点点当规划工具，而非搜索引擎
100 条 Do-计划 query 抽样显示，47% 的 query 是详细的行程规划请求，包含精确日期、人数、预算、约束条件。用户一次性把所有信息给完，期望 AI 输出"可直接执行的行程"。这与综搜的"XX 攻略"三四个字形成本质区别 —— Chat 场景天然承载复杂规划需求。
5. 主站是流量主力，但需求结构和独立端一致
旅游 query 67.1% 来自主站（小红书 App 内的点点入口），32.9% 来自独立端。主站用户"看笔记后自然发问"的行为链路天然适合旅游 —— 看到别人的游记就想问"我也想去，帮我规划一下"。这意味着搜索→点点的引流路径值得重点建设。
6. 基建缺口大，但方向正确：携程数据+地图是刚需
当前 AIM 旅游 session 的工具调用数据无法获取（无独立 tool_call 表），但从 query 内容看，用户对酒店价格(7%)、门票预订(7%)、地图路线(47%)、美食推荐(10%) 有强烈需求。执行文档中携程数据接入、地图路线规划的规划完全契合。

1.2 根因分析：2.06% 是心智问题，不是能力问题

几个支撑判断：

  • 综搜旅游 PV 占比 2.54%（广义口径，日均 ~1955 万 PV），说明小红书整体的旅游流量池并不小
  • 旅游用户的意图质量极高：77.7% 是 Do+Decide，远高于 AIM 大盘的 Do(28.4%)+Decide(11.5%)=39.9%
  • 旅游 session 多轮率 77.4%，说明来了的用户确实在深度使用
  • 但日均仅 ~9,800 UV，对比小红书日活量级，渗透率极低

结论：用户不是来了之后觉得没用走了，而是根本还没想到来。AIM 旅游场景的核心挑战是用户心智建设，不是产品能力提升。

4.5 可能的解释

为什么 22.6% 的旅游 session 只有 1 轮？

  • 固定 sug 点击：用户点了首页的"帮我规划本地温泉3日游"看了一眼就走 —— 好奇心驱动，不是真需求
  • AI 回答不够好：第一轮回答太笼统/太像百科，用户判断"这东西不行"直接离开
  • 纯信息查询：Know 型 query（"XX 签证怎么办"）拿到答案就走，不需要多轮

为什么 15.5% 能聊 10+ 轮？

  • 这批用户是真正在做旅行规划：给完基本信息后，持续追问"第三天换个景点""加一个餐厅推荐""酒店预算降一点"
  • 旅行规划天然是迭代式决策，不是一问一答能解决的
  • 这 15.5% 的用户是产品验证的核心：如果能让更多 1 轮用户转化成 10+轮用户，说明产品能力跟上了

六、值得做什么，不值得做什么

值得做

1. 搜索→AIM 旅游引流路径建设

综搜日均 883 万强旅游意图 PV，AIM 旅游日均仅 ~1.5 万 query。AIO 必玩榜/旅游卡片→点点的引流是投入产出比最高的事。用户已经在搜旅游了，只是不知道来点点问。

2. 携程数据接入 + 地图路线规划

47% 的 query 是详细路线规划，7% 明确提到酒店/门票预订。没有地图和实时数据，AI 只能给出"百度百科级别"的通用建议，无法生成"可执行的行程"。这是产品从"答题机器"升级为"规划工具"的关键。

3. 利用已有笔记聚合能力做"本地推荐"

20% 的 query 是本地/周边游，10% 是美食推荐。这些场景 AIM 已有 note_read / opinion_aggregation 基础能力，搭配 LBS 定位就能快速提供差异化的本地推荐。

有条件地做

4. "带娃/带老人"专项优化

10% 的 query 涉及家庭出游，这些 query 复杂度最高（要考虑年龄/体力/兴趣差异）。如果能做好，是口碑传播的最佳切入点（妈妈群分享），但需要专项数据和知识支撑。

5. 出境游信息补全

抽样中出境游占比不低（马来西亚/泰国/印度/埃及），但出境游需要签证、汇率、当地交通等专项信息。可分阶段，先做东南亚热门线路。

不值得做

6. 不要急着做"AI 追问"来延长对话

旅游 session 多轮率已有 77.4%，不需要通过 AI 主动追问来"拉长对话"。执行文档中"AIM 过于着急给答案，降低用户信任感"的判断是对的 —— 但解决方案不是追问更多，而是第一轮回答要有信息增量（引用真实笔记/评价，而非通用建议）。

7. 不要过度投入独立端 DR 攻略模式

DR 渗透率 2.34% 已平台期，主站未上线。旅游 67% 的流量来自主站 —— 优先在主站做好旅游体验，比优化独立端攻略模式的 ROI 更高。

八、对抗审查

对本报告核心结论的自我质疑和置信度评估：

审查点 1：行程规划是核心需求（41.2%）—— 有多少是泛搜索？中置信

反驳：L2-1 中存在大量泛搜索 query（如"XX旅游攻略"），真实意图可能更接近景点推荐而非完整行程规划。暑期数据天然偏向行程规划。

判断：部分成立。即使保守将 L2-1 中 20-30% 泛搜索移至 L2-6，行程规划大类仍是第一大需求（35%+）。季节性偏差是真实风险。

置信度调整："行程规划是最大需求类别"高置信，"综合行程规划占 24.9%"中置信（实际可能在 18-25% 之间）

审查点 2：多轮率 77.4% vs 另一份报告 90.6% 单轮 —— 口径差异高置信

差异来源：77.4% 多轮率来自 dibp 大盘底表 turn_assembled 的 max(turn_seq)，包含 AI 追问产生的轮次。另一份报告的 90.6% 单轮是只数用户主动发出的 query 条数。两个口径都是事实，回答的是不同问题。

判断：不矛盾。77.4% 说明"旅游场景的 AI 交互深度高"（含 AI 主动追问），90.6% 说明"大多数用户只主动发了 1 条 query"。前者反映产品能力，后者反映用户主动性。

建议：两个数据都有价值。如果要衡量"用户是否愿意持续使用"，应以用户主动 query 数为准。

审查点 3：15.8% 非旅游 query 的影响高置信

反驳：639 条非旅游 query（教育升学、体育赛事、追星行程、购物种草等）被标注为旅游，说明场景路由存在盲区。如果这些流量被路由到旅游 AI 能力，用户体验会受损。

判断：对本报告分析结论不影响（已排除），但对产品体验需关注。

审查点 4：笔记 JSON query 是用户主动还是系统附带？中置信

反驳：约 129 条笔记 JSON query（3.8%）中需区分用户主动引用 vs 系统自动附带笔记内容。后者情况下"UGC 消费场景"可能被高估。

判断：部分成立。但无论哪种情况，笔记消费场景中产生旅游需求是真实的，两者都说明联动价值。"笔记场景与 AI 对话有联动价值"高置信，"用户主动从笔记发起 AI 咨询"中置信。

九、几个需要注意的数据局限

  1. 43.9% 的 query 未被分类 —— scene/intent 模型打标覆盖率仅 56.1%。旅游的真实占比可能高于或低于 3.68%，取决于未分类 query 中旅游的比例。
  2. 数据延迟 4 天 —— 7/3 只能看到 6/29 的数据。近期的产品变更效果无法即时观察。
  3. 6/23-6/25 打标模型异常 —— scene 打标可能在 6/26 才上线或补齐，前 3 天旅游 query 仅约正常值的 1/5。7 天汇总数据会低估真实量级，应以 6/26-6/29 的日均为准。
  4. 性别/年龄画像缺失 —— PI3 敏感字段无权限，无法分析旅游用户的性别和年龄结构。
  5. AIO 旅游数据缺失 —— search_keyword（PI4）权限今日失效，无法做 AIO→AIM 的转化分析。
  6. 工具调用数据缺失 —— 无独立 tool_call 表，skill_id 99.9% 为 NULL。无法量化 search_request / note_read / opinion_aggregation 等工具的覆盖率。
  7. query 抽样有限 —— 热门目的地/主题分析基于 100 条随机抽样，存在统计偏差。结论需要全量分析验证。
  8. L2 标注为 LLM 自动标注 —— 抽样 20 条复核准确率约 95%，约 5% 分类偏差。非人工标注。
  9. 多轮率口径差异 —— 本报告使用 dibp 底表 turn_seq 口径（77.4% 多轮），与 query 级标注口径（90.6% 单轮）不同。详见对抗审查第 2 点。
